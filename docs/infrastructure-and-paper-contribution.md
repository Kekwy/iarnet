# 基础设施部署与论文贡献说明

本文档说明 iarnet 系统中 OSS / STUN / TURN 等基础设施的部署策略，以及从写论文角度如何体现相关工作与创新点。

---

## 一、基础设施部署策略建议

### 1.1 推荐方案：外置成熟服务 + 控制平面只做编排

建议 **OSS、STUN、TURN 均使用成熟外置服务**，控制平面**不实现这些协议本身**，只负责**选用、配置与编排**。

| 组件 | 建议 | 原因 |
|------|------|------|
| **OSS** | 外置（MinIO / 阿里云 OSS / S3 等） | 标准 S3 API，业界通用；论文价值在「怎么用」而非「再实现一遍 OSS」。 |
| **STUN/TURN** | 外置（如 coturn） | NAT 穿透协议复杂、已有成熟实现；论文价值在「如何纳入整体架构、如何配置与评估」。 |
| **控制平面** | 只做编排与策略 | 负责：把 artifact 上传 OSS、生成拉取 URL、在部署请求里下发给 Adapter；配置 STUN/TURN 端点并下发给 Device Agent；可选地做 URL 签名、鉴权、fallback 策略。 |

若把 STUN/TURN 或 OSS **自研并塞进控制平面**：

- 实现成本高，且容易被审稿人视为「重复造轮子」；
- 审稿人更关心的是：**你在调度、Actor 通信、artifact 生命周期上的设计与创新**，而非自研一个 coturn/MinIO。

因此：**基础设施外置，控制平面做「集成与编排」**，既符合工程实践，也便于在论文里把篇幅和贡献点放在核心研究上。

---

## 二、Artifact 拉取策略（设计决策）

系统采用 **pull-on-deploy**（Adapter 收到部署请求后再拉取 artifact），而非控制平面主动推送。

### 2.1 设计理由

- **「就绪」的判定在 Adapter 一侧**：只有 Adapter 知道 artifact 是否已落盘；部署请求到达后再拉，拉完再部署，顺序清晰。
- **控制平面逻辑简单**：只发 `DeployInstanceRequest`（带 `artifact_id` 及拉取地址），不再维护「谁已经传完、谁可以部署」的状态。
- **传输失败与重试好处理**：拉取失败即本次部署失败；重试时再次拉取或命中本地缓存，无需在控制平面维护复杂状态机。
- **同一 artifact 正在拉取则等待**：Adapter 内按 `artifact_id` 去重，多个部署请求需要同一 artifact 时等待同一拉取完成后再部署，避免重复拉取并保证顺序。

### 2.2 与论文的关系

该策略可作为「**方法论/设计决策**」贡献在论文中单独说明，并与「先推送再部署」做对比分析（顺序一致性、失败重试、多 Adapter 扩展性）。

## 三、在论文中体现工作量与创新点

不依赖「自研 OSS/STUN/TURN」，可在以下方面体现工作与创新。

### 3.1 系统架构与组件角色

- 明确画出：**控制平面、OSS、STUN/TURN、Adapter、Device Agent** 的部署关系与数据流。
- 说明**为什么**需要这三类基础设施（artifact 持久化与分发、NAT 穿透），以及**为什么**采用外置而非自研（成熟、可替换、符合实践）。
- 强调：**控制平面不替代这些服务，而是统一配置、发现、下发策略**（例如 ICE 的 STUN/TURN 服务器列表、artifact 拉取 URL 的生成与注入）。

→ 体现**系统设计**上的工作量和清晰度。

### 3.2 Artifact 的「拉取策略」作为设计贡献

- 明确写出采用 **pull-on-deploy** 及设计理由。
- 可选：与「先推送再部署」做**对比分析**（顺序一致性、失败重试、多 Adapter 扩展性），用一小节或一段说清楚。

→ **方法论/设计决策**上的贡献。

### 3.3 集成与编排设计

- **Artifact 路径**：控制平面（或构建流水线）将构建产物上传 OSS → 生成带过期时间的访问方式（如预签名 URL）→ 在 `DeployInstanceRequest` 中把「拉取地址 + artifact_id」交给 Adapter；Adapter 按需拉取、按 artifact_id 去重/等待。
- **ICE 路径**：控制平面配置 STUN/TURN 的 endpoint，通过信令或配置下发给 Device Agent；Agent 在 ICE 协商时使用这些服务器；可简述 fallback（先 STUN，失败再用 TURN）。

→ 体现**集成方案**和**可部署性**。

### 3.4 实验与评估

- **Artifact**：对比或分析 **pull vs push** 的首次部署延迟、多节点并发部署时的拉取去重效果，或不同 OSS 部署位置（同域/跨域）对拉取延迟的影响。
- **NAT 穿透**：在受控环境中（如不同 NAT 类型、有无 TURN）统计 **仅 STUN / STUN+TURN** 下的建链成功率、延迟，说明「引入外置 STUN/TURN 后，本系统在真实网络下的可达性」。
- 若有**端到端工作流**：可给出「含 artifact 拉取 + 调度 + Actor 部署 + P2P 建链」的时延分解，突出 artifact 拉取只占其中一部分且可控。

→ **实验性贡献**，直接回答「做了什么、效果如何」。

### 3.5 可选的「自主实现」部分

在不偏离主线的前提下，可在控制平面侧做一层薄服务，增加「自主实现」的篇幅：

- **Artifact 元数据与 URL 服务**：控制平面上传 OSS 后，在自有 DB 或配置中记录 `(artifact_id, OSS key, 版本)`，按需生成预签名 URL 并下发给 Adapter；可加权限、租户、过期策略。论文中可写为「artifact 生命周期管理与安全拉取策略」。
- **ICE 参数与策略配置**：控制平面根据设备/拓扑选择不同的 STUN/TURN 组合或优先级，并下发给 Device Agent（仍用外置 STUN/TURN 服务器）。论文中可写为「面向多设备的 NAT 穿透策略配置」。

## 四、论文结构建议

- **系统架构图**：明确标出 OSS、STUN、TURN 为**外置组件**，控制平面、Adapter、Device Agent 为**自研组件**，并用短句说明「本系统采用外置 OSS/STUN/TURN，控制平面负责编排与策略下发」。
- **专门小节**（如「Artifact 分发与拉取」「NAT 穿透与信令」）：
  - 先写**设计选择**（pull-on-deploy、外置基础设施）；
  - 再写**集成方式**（谁上传 OSS、谁生成 URL、谁配置 STUN/TURN、谁拉取、谁建链）；
  - 最后用**实验**说明效果（延迟、成功率、可扩展性）。
- **相关工作/对比**：与「自建全套基础设施」或「仅用单一中心化传输」的方案对比，突出本系统在**架构清晰度、可部署性、artifact 与 NAT 策略**上的设计。

---

## 五、小结

- **推荐**：OSS、STUN、TURN 均使用成熟外置服务；控制平面负责**编排、配置、URL 生成与下发、策略**，不实现 OSS/STUN/TURN 协议本身。
- **论文中的工作与创新**集中在：
  1. 整体系统架构与组件角色；
  2. artifact 的 pull-on-deploy 设计与理由；
  3. 与外置 OSS/STUN/TURN 的集成与编排设计；
  4. 上述设计的实验评估（延迟、成功率、可扩展性）；
  5. 可选的「artifact 元数据/URL 与 ICE 策略」的自主设计与实现。

这样既符合工程实践，又能在论文中清晰体现工作量和创新点，而不会把重点分散到「重做一遍 OSS/STUN/TURN」上。
